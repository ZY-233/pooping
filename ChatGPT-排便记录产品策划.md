# 二、产品目标

V1 只验证三件事：
1. 用户是否愿意持续记录
2. 用户是否喜欢简单统计反馈
3. 好友绑定和轻互动是否能提升留存

# 三、V1 核心功能

请围绕下面这些功能开发，不要擅自扩展成复杂产品：

## 1. 快速记录
用户可以快速记录一次排便，字段包括：
- 时间（必填，默认当前时间，可修改）
- 形状（选填）
- 感受（选填）
- 备注（选填，短文本）

### 形状选项
- 偏干
- 正常
- 偏稀

### 感受选项
- 很顺
- 一般
- 有点费劲
- 很急

## 2. 首页摘要
首页需要展示：
- 今日是否已记录
- 最近一次记录时间
- 本月次数
- 本周次数
- 好友状态卡（如果已绑定噗友）

## 3. 日历 / 统计
需要一个日历页，支持：
- 查看某个月的记录分布
- 显示本月累计次数
- 显示每天是否有记录
- 查看某天的记录列表
- 进入记录详情页

## 4. 记录详情
支持：
- 查看记录详情
- 编辑记录
- 删除记录（带二次确认）

## 5. 噗友绑定
V1 只支持 1 对 1 噗友关系，支持：
- 生成邀请链接或邀请码
- 接受邀请
- 查看当前绑定状态
- 解除噗友关系

## 6. 噗友轻互动
当噗友记录后，可以看到轻提示，并可以发送轻回应：
- 顺顺顺
- 鼓掌
- 爱心
- 喝水提醒

## 7. 隐私设置
支持设置噗友可见范围：
- 共享“今天是否记录”
- 共享记录时间
- 共享形状信息
- 共享备注内容

默认只共享第一项。

## 8. 提醒设置
支持：
- 开关每日提醒
- 设置提醒时间
- 提醒风格：温柔型 / 可爱型

V1 可以先只保存设置，不必完整实现复杂推送系统。

## 9. 非医疗说明
需要独立页面或说明区，明确：
- 本产品仅用于日常记录
- 不提供医疗诊断、疾病判断或治疗建议

# 九、权限与安全要求

这个产品是私密记录产品，权限要严格控制。

请设计 Supabase RLS 策略，满足以下原则：

## 用户自己的数据
- 用户只能读写自己的 profile
- 用户只能读写自己的 poop_records
- 用户只能修改自己的 settings

## 噗友关系
- 只有关系双方可以读取 relation
- 只有关系相关用户可以解除绑定
- 只有用户自己可以修改自己的共享设置

## 噗友可见数据
不要让前端直接读取对方完整 poop_records。
请设计一层服务端逻辑 / 受控查询，只返回：
- 今天是否记录
- 如果已授权，可返回时间摘要
- 如果已授权，可返回形状
- 如果已授权，可返回备注

也就是说，要有“friend visibility / shared summary”逻辑，而不是直接把主记录表裸给前端。
